# 3.1 – Introducción a la Mensajería Distribuida

---

## 🎯 Objetivo

Comprender el papel de la **mensajería distribuida** dentro de una arquitectura de microservicios, y cómo los servicios pueden comunicarse **de manera asíncrona** usando *message brokers* como **RabbitMQ o Kafka** para desacoplar procesos, mejorar la resiliencia y escalar el sistema de forma más eficiente.

---

## 🧩 Contexto

En los temas anteriores vimos cómo los servicios pueden comunicarse entre sí de forma **síncrona** (REST, HTTP).
Ese modelo es útil, pero tiene una limitación:

> cuando un servicio falla, **todo el flujo se detiene**.

La **mensajería distribuida** surge como respuesta a ese problema: los servicios dejan de depender de la disponibilidad inmediata de otros y se comunican **a través de mensajes**, de manera **asíncrona y desacoplada**.

---

## ⚙️ Concepto de mensajería distribuida

En este modelo, los servicios no se llaman directamente.
En su lugar:

1. **Un servicio publica un mensaje** (por ejemplo, “pedido_creado”) en un intermediario.
2. **El broker almacena y distribuye** ese mensaje de forma fiable.
3. **Otros servicios suscritos** lo reciben y lo procesan cuando estén disponibles.

Este intermediario se llama **message broker**.
Ejemplos populares:

* 🐇 **RabbitMQ**
* 🦍 **Kafka**
* ☁️ **Amazon SQS**
* 🔷 **Azure Service Bus**

---

## 🔗 Ventajas frente a REST

| Aspecto                    | REST (síncrono)                               | Mensajería (asíncrono)             |
| -------------------------- | --------------------------------------------- | ---------------------------------- |
| **Acoplamiento temporal**  | Alto (debe estar disponible el otro servicio) | Bajo (se puede procesar más tarde) |
| **Tolerancia a fallos**    | Baja                                          | Alta                               |
| **Latencia**               | Inmediata                                     | Diferida                           |
| **Escalabilidad**          | Limitada                                      | Natural (procesos en paralelo)     |
| **Modelo de comunicación** | Petición → Respuesta                          | Publicación → Suscripción          |

---

## 🧱 Arquitectura básica

```
[Productor] → (Publica mensaje) → [Broker de Mensajes] → (Entrega) → [Consumidor]
```

Cada mensaje representa un **evento de negocio**.
Ejemplo:

* `pedido_creado`
* `usuario_registrado`
* `producto_actualizado`

---

## 🧩 Ejemplo funcional con RabbitMQ

Imaginemos un flujo de **pedidos y facturación**:

* El servicio `orders` crea un pedido y **publica un evento** `pedido_creado`.
* El servicio `billing` **escucha ese evento** y genera una factura.

Asumiremos que el broker RabbitMQ ya está funcionando (puede estar definido en `docker-compose.yml`).

---

### 🐇 Productor – Servicio de Pedidos

```js
import amqp from "amqplib";

async function publicarPedido(pedido) {
  const conn = await amqp.connect("amqp://rabbitmq");
  const ch = await conn.createChannel();

  const cola = "pedidos_creados";
  await ch.assertQueue(cola, { durable: true });

  const mensaje = JSON.stringify(pedido);
  ch.sendToQueue(cola, Buffer.from(mensaje));

  console.log(`📤 Pedido publicado: ${pedido.id}`);
  await ch.close();
  await conn.close();
}

// Simulación de creación de pedido
const nuevoPedido = { id: Date.now(), user: "Ana", total: 42.9 };
publicarPedido(nuevoPedido);
```

---

### 💰 Consumidor – Servicio de Facturación

```js
import amqp from "amqplib";

async function escucharPedidos() {
  const conn = await amqp.connect("amqp://rabbitmq");
  const ch = await conn.createChannel();

  const cola = "pedidos_creados";
  await ch.assertQueue(cola, { durable: true });

  console.log("📥 Esperando eventos de 'pedidos_creados'...");

  ch.consume(cola, msg => {
    const pedido = JSON.parse(msg.content.toString());
    console.log(`🧾 Generando factura para pedido ${pedido.id} (${pedido.user})`);
    ch.ack(msg);
  });
}

escucharPedidos();
```

---

### 🔍 Qué observamos aquí

* El **servicio de pedidos no depende del de facturación**.
  Publica el mensaje y sigue trabajando.
* Si `billing` está caído, el mensaje **permanece en la cola** hasta que se reconecte.
* Es posible tener **múltiples consumidores** escuchando la misma cola o distintas.

👉 Con esto hemos logrado **desacoplar temporalmente los servicios**.

---

## 🧠 Conceptos clave

| Concepto                    | Descripción                                                              |
| --------------------------- | ------------------------------------------------------------------------ |
| **Productor (Publisher)**   | Envía mensajes al broker.                                                |
| **Consumidor (Subscriber)** | Recibe y procesa mensajes.                                               |
| **Cola (Queue)**            | Estructura FIFO donde se almacenan los mensajes hasta ser consumidos.    |
| **Exchange (enrutador)**    | Componente que distribuye mensajes según reglas (fanout, direct, topic). |
| **Acknowledgment (ack)**    | Confirmación de que un mensaje fue procesado con éxito.                  |

---

## 🕸️ Modos de enrutamiento comunes

| Tipo de Exchange | Descripción                                   | Ejemplo de uso                      |
| ---------------- | --------------------------------------------- | ----------------------------------- |
| **fanout**       | Envía el mensaje a todas las colas enlazadas. | Broadcast de eventos globales.      |
| **direct**       | Usa una clave de enrutamiento exacta.         | Un mensaje va a una cola concreta.  |
| **topic**        | Usa patrones de enrutamiento (comodines).     | Filtrar eventos por tipo o dominio. |

Ejemplo con *topic*:

```js
ch.publish("eventos", "pedido.creado", Buffer.from(JSON.stringify(pedido)));
```

---

## 🧩 Integración con patrones de resiliencia

La mensajería se complementa perfectamente con los patrones vistos antes:

| Patrón              | Cómo se combina                                                             |
| ------------------- | --------------------------------------------------------------------------- |
| **Retry**           | RabbitMQ puede reintentar automáticamente entregar mensajes.                |
| **Timeout**         | Los consumidores pueden limitar su tiempo de procesamiento.                 |
| **Circuit Breaker** | Si el broker está caído, el publicador corta la comunicación temporalmente. |
| **Fallback**        | Si no se puede enviar el mensaje, se guarda en una “cola de espera” local.  |

---

## 🔧 Comparativa práctica: REST vs Mensajería

| Situación                           | REST (Síncrono)                   | RabbitMQ (Asíncrono)                                 |
| ----------------------------------- | --------------------------------- | ---------------------------------------------------- |
| El servicio destino está caído      | La petición falla                 | El mensaje se encola y se procesará más tarde        |
| Escalado horizontal                 | Complejo, necesita load balancing | Natural, los consumidores pueden aumentar            |
| Auditoría de eventos                | Requiere logs externos            | El broker conserva trazabilidad de mensajes          |
| Coordinación entre varios servicios | Multiplica las llamadas           | Cada servicio se suscribe a los eventos que necesita |

---

## 💬 Analogía

> Imagina que REST es **una llamada telefónica**:
> necesitas que la otra persona te conteste ahora mismo.
>
> En cambio, la mensajería es **como enviar un correo electrónico**:
> lo mandas, queda entregado al buzón, y el receptor lo leerá cuando pueda.
>
> Así logras **desacoplar el tiempo** de emisión y recepción.

---

## 🔚 Conclusión

La mensajería distribuida representa un **cambio de paradigma**:
los servicios dejan de comunicarse “cara a cara” y comienzan a hacerlo **por eventos**.

Esto permite:

* Tolerar fallos sin interrupciones.
* Escalar consumidores según demanda.
* Implementar flujos complejos mediante **event-driven architectures**.

En los siguientes apartados (3.2 y 3.3) profundizaremos en **brokers y enrutamiento** y en **coreografía frente a orquestación** para coordinar flujos entre servicios.
Los patrones **CQRS y Event Sourcing** se estudian en el módulo 4.

# 2.1 – Comunicación entre servicios

---

## 🎯 Objetivo

Comprender cómo los microservicios se comunican entre sí dentro de una arquitectura distribuida, tanto de forma **síncrona (REST)** como **asíncrona (mensajería)**, y analizar los efectos que cada enfoque tiene sobre el rendimiento, la independencia y la resiliencia del sistema.

---

## 🧩 Contexto

En una arquitectura de microservicios, cada servicio es **independiente y autónomo**, pero rara vez funciona aislado.
El valor real surge cuando varios servicios **colaboran entre sí** para cubrir un flujo de negocio completo: usuarios, pedidos, inventario, pagos…

A diferencia del monolito, donde los módulos comparten memoria o llamadas de función, los microservicios **se comunican a través de la red**.
Esto introduce nuevas posibilidades —y nuevos retos— en términos de latencia, disponibilidad y consistencia.

---

## 🔗 Tipos de comunicación

Los microservicios pueden intercambiar información de dos formas fundamentales:

| Tipo          | Características                                                                       | Ejemplo típico                    |
| ------------- | ------------------------------------------------------------------------------------- | --------------------------------- |
| **Síncrona**  | El servicio A llama al servicio B y **espera su respuesta**.                          | REST, gRPC                        |
| **Asíncrona** | El servicio A **envía un mensaje** y continúa su ejecución; B lo procesará más tarde. | RabbitMQ, Kafka, colas de eventos |

---

## ⚙️ Comunicación síncrona (REST)

La comunicación síncrona es la más intuitiva: un servicio **invoca una API HTTP** de otro servicio, espera la respuesta y continúa.
Este modelo se utiliza cuando la información es inmediata o esencial para completar la operación.

---

### 🔹 Ejemplo conceptual: servicio de pedidos que consulta usuarios

Imaginemos dos microservicios:

* `users-service`: gestiona los usuarios registrados.
* `orders-service`: gestiona los pedidos y necesita conocer los datos del usuario que los realiza.

```js
// Servicio de pedidos
import axios from "axios";

async function crearPedido(userId, productos) {
  // Llamada HTTP a otro microservicio
  const usuario = await axios.get(`http://users-service:3001/users/${userId}`);

  const pedido = {
    id: Date.now(),
    user: usuario.data.name,
    productos,
    total: productos.reduce((s, p) => s + p.precio, 0)
  };

  console.log(`🧾 Pedido creado para ${pedido.user}: ${pedido.total} €`);
  return pedido;
}
```

**Qué ocurre:**

* El servicio `orders` realiza una llamada a `users` y **espera la respuesta**.
* Si `users-service` está caído o responde lento, el flujo completo se bloquea.

Este patrón es simple pero introduce **acoplamiento temporal**: el éxito de una operación depende de la disponibilidad del servicio remoto.

---

### 🔍 Implicaciones del modelo síncrono

| Ventajas                                           | Inconvenientes                                                          |
| -------------------------------------------------- | ----------------------------------------------------------------------- |
| Comunicación directa, fácil de entender.           | Acoplamiento fuerte: un servicio depende de la disponibilidad del otro. |
| Ideal para operaciones de lectura o validación.    | Introduce latencia y riesgo de fallo en cascada.                        |
| Compatible con herramientas estándar (HTTP, JSON). | Requiere manejo de errores, timeouts y reintentos.                      |

Ejemplo de manejo básico de error:

```js
try {
  const res = await axios.get("http://users-service/users");
  return res.data;
} catch (error) {
  console.error("❌ No se pudo conectar con users-service:", error.message);
  return { error: "Servicio temporalmente no disponible" };
}
```

## 🕊️ Comunicación asíncrona (mensajería)

Cuando los servicios deben **intercambiar información sin depender de la respuesta inmediata del otro**, se utiliza **mensajería asíncrona** mediante un *broker* (intermediario).

En este modelo:

1. Un servicio **publica un mensaje** en una cola o tópico.
2. Otro servicio **suscrito** lo recibe y lo procesa más tarde.
3. No existe bloqueo entre ambos: cada uno trabaja a su ritmo.

---

### 🔹 Ejemplo conceptual: RabbitMQ como bus de eventos

Imaginemos que, cada vez que se crea un pedido, queremos notificar a otros servicios (como facturación o envío).
No necesitamos respuesta inmediata; basta con **publicar un evento “pedido_creado”** que otros escuchen.

#### Emisor (servicio de pedidos)

```js
import amqp from "amqplib";

async function publicarPedido(pedido) {
  const conn = await amqp.connect("amqp://localhost");
  const canal = await conn.createChannel();
  const cola = "pedidos_creados";

  await canal.assertQueue(cola, { durable: true });
  canal.sendToQueue(cola, Buffer.from(JSON.stringify(pedido)));

  console.log(`📤 Pedido publicado: ${pedido.id}`);
  await canal.close();
  await conn.close();
}
```

#### Receptor (servicio de facturación)

```js
import amqp from "amqplib";

async function escucharPedidos() {
  const conn = await amqp.connect("amqp://localhost");
  const canal = await conn.createChannel();
  const cola = "pedidos_creados";

  await canal.assertQueue(cola, { durable: true });
  console.log("📥 Esperando eventos de pedidos...");

  canal.consume(cola, (msg) => {
    const pedido = JSON.parse(msg.content.toString());
    console.log(`🧾 Generando factura para pedido ${pedido.id}`);
    canal.ack(msg);
  });
}
```

---

### 🔍 Qué aporta este modelo

* **Desacoplamiento temporal:** el emisor no depende del consumidor.
* **Resiliencia:** si un servicio está caído, el mensaje queda en cola.
* **Escalabilidad natural:** múltiples consumidores pueden procesar en paralelo.

Sin embargo, también introduce **complejidad operativa**:

* Hay que gestionar el *broker* (RabbitMQ, Kafka, etc.).
* Se requiere control de reintentos, duplicados y orden de mensajes.
* No hay respuesta inmediata; la confirmación es eventual.

---

## 🧠 Comparativa entre comunicación síncrona y asíncrona

| Aspecto                  | Síncrona (REST)                  | Asíncrona (mensajería)             |
| ------------------------ | -------------------------------- | ---------------------------------- |
| **Dependencia temporal** | Alta (bloqueante)                | Baja (no bloqueante)               |
| **Latencia percibida**   | Inmediata                        | Diferida                           |
| **Acoplamiento**         | Fuerte                           | Débil                              |
| **Complejidad técnica**  | Menor                            | Mayor                              |
| **Escalabilidad**        | Limitada por el servicio remoto  | Natural con colas                  |
| **Uso típico**           | Validaciones, consultas directas | Eventos, notificaciones, pipelines |

---

## 🧩 Buenas prácticas

1. **Combinar ambos modelos** según el caso de uso:

   * REST para interacciones inmediatas.
   * Mensajería para procesos desacoplados o notificaciones.

2. **Documentar los contratos** de comunicación (endpoints o tipos de mensaje).

3. **Asegurar idempotencia** en los consumidores asíncronos para evitar duplicados.

4. **Implementar trazabilidad** (correlation IDs) para seguir flujos entre servicios.

5. **Simular fallos** de red o caídas para probar la resiliencia de ambos modelos.

---

## 💬 Resumen

* La comunicación entre microservicios puede ser **síncrona o asíncrona**.
* La **síncrona** es inmediata pero dependiente de la disponibilidad del otro servicio.
* La **asíncrona** permite independencia temporal mediante un *broker* (RabbitMQ/Kafka).
* Entender cuándo usar cada modelo es esencial para diseñar sistemas escalables y resilientes.
* La combinación de ambos patrones es lo que da flexibilidad real a una arquitectura distribuida moderna.

---

## 🔚 Conclusión

La comunicación entre servicios define el “sistema nervioso” de los microservicios.
Su diseño impacta directamente en la **agilidad, escalabilidad y resiliencia** del sistema.
En los siguientes apartados profundizaremos en los mecanismos que ayudan a **gestionar esta comunicación de forma automática y segura**, como el **Service Discovery** y los **patrones de resiliencia**.

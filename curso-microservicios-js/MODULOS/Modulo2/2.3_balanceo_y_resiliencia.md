el la# 2.3 – Balanceo y resiliencia

---

## 🎯 Objetivo

Comprender cómo las arquitecturas de microservicios gestionan la **distribución de carga entre instancias** (balanceo) y cómo aplican **patrones de resiliencia** para sobrevivir a fallos parciales sin comprometer la disponibilidad global del sistema.

---

## 🧩 Contexto

En un entorno monolítico tradicional, la aplicación se ejecuta como un único proceso:
si falla, el sistema completo se detiene.

En cambio, en una arquitectura distribuida:

* cada microservicio puede tener **múltiples instancias** activas,
* y la comunicación entre ellas puede **redirigirse dinámicamente** según disponibilidad o carga.

A esto se le llama **balanceo de carga** (*Load Balancing*).
Pero incluso con balanceo, los fallos son inevitables. Por ello, necesitamos estrategias de **resiliencia**, es decir, la capacidad del sistema de **mantenerse funcional ante errores**.

---

## ⚖️ Balanceo de carga

El **balanceo** distribuye las peticiones de entrada entre varias instancias de un mismo servicio para:

* mejorar el rendimiento,
* evitar sobrecargas,
* y garantizar alta disponibilidad.

### 🔹 Ejemplo conceptual

Supón que tienes tres instancias del servicio `users-service`:

```
users-service-1 → localhost:3001
users-service-2 → localhost:3002
users-service-3 → localhost:3003
```

Un balanceador se encarga de decidir a cuál enviar cada petición.
Puede ser un componente dedicado (NGINX, HAProxy) o parte del propio cliente.

---

### 🔸 Ejemplo funcional de balanceo simple (client-side)

Un cliente puede mantener un pequeño “pool” de instancias y alternar las peticiones:

```js
import axios from "axios";

const instancias = [
  "http://localhost:3001",
  "http://localhost:3002",
  "http://localhost:3003"
];

let indice = 0;

async function obtenerUsuarios() {
  const url = instancias[indice];
  indice = (indice + 1) % instancias.length; // round-robin
  const res = await axios.get(`${url}/users`);
  return res.data;
}

// Cada llamada va a una instancia diferente
```

Este patrón se llama **Round Robin**, y es una forma sencilla de distribuir tráfico sin necesidad de un balanceador externo.

En entornos reales, esta función la cumplen:

* **Balanceadores externos**: NGINX, AWS ELB, Traefik.
* **Orquestadores**: Kubernetes Services reparten las peticiones automáticamente.
* **API Gateways**: manejan routing y distribución de carga de forma centralizada.

---

## 🧱 Resiliencia

Incluso con balanceo, los servicios pueden fallar:
una instancia puede estar caída, sobrecargada o devolver errores intermitentes.

La **resiliencia** busca **evitar que los fallos locales se propaguen** al resto del sistema.

Los patrones más usados son:

1. **Retry (reintento)**
2. **Timeout (tiempo máximo de espera)**
3. **Circuit Breaker (disyuntor)**
4. **Bulkhead (compartimentos)**
5. **Fallback (respuesta alternativa)**

Veamos cómo funcionan en la práctica.

---

### 🔹 1. Retry – volver a intentar

Cuando una llamada falla por un error temporal, puede ser suficiente **reintentarla una o dos veces**.

```js
import axios from "axios";

async function llamadaConReintento(url, intentos = 3) {
  for (let i = 0; i < intentos; i++) {
    try {
      const res = await axios.get(url);
      return res.data;
    } catch (e) {
      console.log(`⚠️ Intento ${i + 1} fallido`);
      await new Promise(r => setTimeout(r, 500)); // espera medio segundo
    }
  }
  throw new Error("❌ No se pudo conectar tras varios intentos");
}
```

✅ **Útil para errores transitorios** (latencia, conexiones temporales).

---

### 🔹 2. Timeout – no esperar indefinidamente

Si un servicio tarda demasiado en responder, es preferible **interrumpir la petición** y continuar con una alternativa o un error controlado.

```js
import axios from "axios";

const cliente = axios.create({ timeout: 2000 }); // 2 segundos máximo

try {
  const res = await cliente.get("http://slow-service:3000/data");
  console.log(res.data);
} catch (error) {
  console.error("⏱️ Tiempo de espera excedido");
}
```

Esto evita que un servicio lento **bloquee al resto del sistema**.

---

### 🔹 3. Circuit Breaker – cortar para proteger

El patrón **Circuit Breaker** (disyuntor) evita sobrecargar un servicio que ya está fallando:
si detecta demasiados errores consecutivos, **interrumpe las llamadas durante un tiempo**.

```js
import CircuitBreaker from "opossum";
import axios from "axios";

async function llamadaRemota() {
  const res = await axios.get("http://users-service:3001/users");
  return res.data;
}

const breaker = new CircuitBreaker(llamadaRemota, {
  timeout: 3000,
  errorThresholdPercentage: 50,
  resetTimeout: 5000
});

breaker.on("open", () => console.log("🚨 Circuito abierto – fallos detectados"));
breaker.on("halfOpen", () => console.log("⚠️ Probando recuperación"));
breaker.on("close", () => console.log("✅ Circuito cerrado – servicio estable"));

const usuarios = await breaker.fire();
```

✅ Cuando el circuito está **abierto**, las peticiones fallan rápidamente sin intentos innecesarios.
Después de un tiempo, pasa a **half-open** y prueba si el servicio se ha recuperado.

---

### 🔹 4. Fallback – respuesta alternativa

En lugar de fallar por completo, un servicio puede **devolver una respuesta por defecto** si el origen no responde.

```js
async function obtenerUsuariosSeguros() {
  try {
    const res = await axios.get("http://users-service/users");
    return res.data;
  } catch {
    console.log("🟡 Usando datos en caché por fallo del servicio");
    return [{ id: 0, name: "Usuario temporal" }];
  }
}
```

✅ Esto mantiene el sistema operativo, incluso con datos parciales.

---

### 🔹 5. Bulkhead – compartimentos aislados

Inspirado en la ingeniería naval, este patrón divide los recursos (como hilos o conexiones) para evitar que **un fallo en un servicio afecte a los demás**.

Ejemplo conceptual:

* Se usa una **cola separada** o **pool de conexiones** por servicio.
* Si una cola se satura, solo se bloquea ese flujo, no todo el sistema.

```js
// Ejemplo conceptual
const poolUsuarios = new ConnectionPool({ size: 5 });
const poolPedidos = new ConnectionPool({ size: 5 });
```

---

## 💬 Combinación práctica de resiliencia

Los patrones anteriores suelen **trabajar juntos**.
Un flujo típico en microservicios podría verse así:

```
retry → timeout → circuit breaker → fallback
```

Cada capa añade un nivel de protección.
Si el servicio no responde, el sistema no se detiene:
reintenta, corta, y devuelve algo útil o un mensaje controlado.

---

## 🧠 Resumen visual

| Patrón              | Propósito                                   | Resultado                        |
| ------------------- | ------------------------------------------- | -------------------------------- |
| **Retry**           | Volver a intentar operaciones fallidas      | Manejo de fallos temporales      |
| **Timeout**         | Limitar el tiempo de espera                 | Evita bloqueos prolongados       |
| **Circuit Breaker** | Interrumpir llamadas repetidamente fallidas | Protege al sistema de sobrecarga |
| **Fallback**        | Respuesta alternativa                       | Mantiene experiencia parcial     |
| **Bulkhead**        | Aislar recursos                             | Previene efectos en cascada      |

---

## ⚙️ Balanceo + resiliencia en conjunto

En un entorno real:

* **El balanceador** reparte carga entre instancias activas.
* **Los mecanismos de resiliencia** gestionan las fallas o lentitud de alguna de ellas.
* **Los registros de salud (health checks)** informan al balanceador qué instancias deben recibir tráfico.

Ejemplo ilustrativo:

```
Cliente → Gateway → Load Balancer → [Instancias saludables]
                                ↘︎ Circuit Breaker / Retry / Timeout
```

---

## 🔚 Conclusión

El balanceo de carga asegura la **distribución eficiente del tráfico**,
y la resiliencia garantiza la **continuidad del servicio ante fallos**.

Juntos conforman el núcleo de la **robustez operativa** en una arquitectura de microservicios.
Sin ellos, cualquier error local podría desencadenar una caída global.

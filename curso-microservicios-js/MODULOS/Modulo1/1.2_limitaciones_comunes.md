# 1.2 – Limitaciones comunes: escalado, acoplamiento y mantenimiento

---

## 🎯 Objetivo

Identificar las **principales limitaciones de las arquitecturas monolíticas** y comprender por qué se vuelven problemáticas cuando las aplicaciones crecen en funcionalidad, tráfico o equipos de desarrollo.

---

## 🧩 Introducción

Una aplicación monolítica **funciona muy bien al inicio del proyecto**, pero con el tiempo surgen **problemas estructurales y operativos**.
Estas limitaciones no son evidentes en los primeros meses, pero se hacen críticas cuando:

* la base de código crece,
* aumenta el número de desarrolladores,
* se necesita escalar solo una parte del sistema.

---

## ⚙️ 1. Escalado limitado

En un monolito, **todos los componentes se ejecutan en el mismo proceso**.

Cuando un módulo necesita más capacidad (por ejemplo, el de pedidos), **no se puede escalar solo él**.
Hay que duplicar **toda la aplicación**, aunque el resto de módulos no lo necesiten.

### 🔸 Ejemplo visual

```
[Servidor 1]
 ├── /users
 ├── /orders
 ├── /inventory

Escalado monolítico ⇒ duplicar todo:
[Servidor 2]
 ├── /users
 ├── /orders
 ├── /inventory
```

➡️ Resultado: consumo innecesario de recursos, despliegues más lentos y costes más altos.

### 🔸 En Node.js

```bash
# No se puede ejecutar solo un módulo de forma independiente:
node app.js
```

La app completa se levanta aunque solo necesites aumentar `/orders`.

---

## 🧩 2. Acoplamiento fuerte

En los monolitos, los módulos están **estrechamente interconectados**:

* Comparten la misma base de datos.
* Se llaman entre sí directamente.
* Usan los mismos modelos y librerías.

### 🔸 Ejemplo de dependencia cruzada

```js
// userService.js
import { createOrder } from './orderService.js';

export function registerUser(name) {
  const user = { name };
  createOrder(user); // dependencia directa
}
```

Si `orderService` cambia, `userService` puede dejar de funcionar.

### 🔸 Consecuencias

* **Dificultad para probar**: hay que levantar toda la app.
* **Errores en cascada**: un fallo en un módulo puede tumbar el sistema completo.
* **Difícil actualización tecnológica**: cambiar una librería afecta a todo el código.

---

## 🧰 3. Mantenimiento complejo

Con el crecimiento del sistema, el código se vuelve **difícil de entender y modificar**.
Los desarrolladores pierden la visión global, y los cambios en un módulo pueden generar efectos colaterales inesperados.

### 🔸 Ejemplo real

> Una aplicación de facturación con cientos de rutas Express, servicios y modelos compartidos.
> Cualquier cambio en “clientes” puede romper “pedidos” o “reportes”.

### 🔸 Síntomas típicos

* Commit que rompe funciones no relacionadas.
* Merge conflicts constantes.
* Despliegues que requieren pruebas masivas antes de subir a producción.

---

## 🚧 Otros problemas derivados

| Área                     | Problema                                                           |
| ------------------------ | ------------------------------------------------------------------ |
| **Despliegue**           | Requiere recompilar todo el artefacto incluso por un cambio menor. |
| **Escalado de equipos**  | Equipos grandes pisan código ajeno. Dificulta el trabajo paralelo. |
| **Fiabilidad**           | Un fallo en un módulo (ej. pagos) puede bloquear toda la app.      |
| **Tiempos de respuesta** | El crecimiento del código ralentiza el arranque y las pruebas.     |

---

## 🧠 Resumen visual

| Limitación            | Causa raíz                   | Consecuencia                        |
| --------------------- | ---------------------------- | ----------------------------------- |
| Escalado global       | Un único bloque de ejecución | Coste y rigidez operativa           |
| Acoplamiento fuerte   | Dependencias entre módulos   | Riesgo alto de errores cruzados     |
| Mantenimiento difícil | Código masivo y compartido   | Lentitud en desarrollo y despliegue |

---

## 💡 Reflexión docente

> Las arquitecturas monolíticas **se comportan como un único organismo**: si una parte enferma, **todo el cuerpo se resiente**.
> Por eso, en la evolución hacia microservicios se busca **independencia funcional y despliegue desacoplado**, para aislar errores, escalar selectivamente y facilitar la evolución del sistema.

---

## 🔚 Conclusión

Las limitaciones del monolito —**escalado global, acoplamiento fuerte y mantenimiento complejo**— son las razones principales por las que surgieron **las arquitecturas distribuidas y los microservicios**, tema que se abordará a continuación en el punto **1.3 – Introducción al paradigma distribuido**.

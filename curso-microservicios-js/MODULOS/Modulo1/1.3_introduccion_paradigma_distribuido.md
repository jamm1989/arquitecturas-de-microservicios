# 1.3 – Introducción al paradigma distribuido

---

## 🎯 Objetivo

Comprender qué es una **arquitectura distribuida**, cómo se diferencia de un monolito y por qué representa el paso natural hacia los **microservicios** en sistemas modernos que requieren escalabilidad, resiliencia y agilidad en el desarrollo.

---

## 🧩 ¿Qué es una arquitectura distribuida?

Una **arquitectura distribuida** es aquella en la que **las funciones de una aplicación están separadas en componentes independientes** que se ejecutan en **procesos o servidores distintos**, pero que **cooperan entre sí mediante comunicación por red** (HTTP, gRPC, mensajería, etc.).

👉 En lugar de tener **una única aplicación que hace todo**, tenemos **varias piezas más pequeñas**, cada una responsable de una tarea específica.

---

## ⚙️ Ejemplo visual

**Monolito (todo junto):**

```
[ APP ÚNICA ]
 ├── /users
 ├── /orders
 ├── /inventory
 └── /billing
```

**Arquitectura distribuida (cada módulo independiente):**

```
[ users-service ] → gestiona usuarios
[ orders-service ] → gestiona pedidos
[ inventory-service ] → controla stock
[ billing-service ] → maneja pagos
```

Cada servicio tiene su propio ciclo de vida:

* Se despliega por separado.
* Puede tener su propia base de datos.
* Escala de manera independiente.

---

## 🔗 Comunicación entre servicios

Como los servicios ya no viven en el mismo proceso, deben **comunicarse por red**.

### 🔸 Ejemplo con Node.js (comunicación REST)

**Servicio de usuarios (`users-service`):**

```js
// users-service/index.js
import express from 'express';
const app = express();

app.get('/users', (req, res) => {
  res.json([{ id: 1, name: 'Ana' }]);
});

app.listen(3001, () => console.log('Users service en puerto 3001'));
```

**Servicio de pedidos (`orders-service`):**

```js
// orders-service/index.js
import express from 'express';
import axios from 'axios';
const app = express();

app.get('/orders', async (req, res) => {
  const { data: users } = await axios.get('http://localhost:3001/users');
  res.json([{ orderId: 100, user: users[0].name, total: 59.90 }]);
});

app.listen(3002, () => console.log('Orders service en puerto 3002'));
```

➡️ Aquí `orders-service` **consume la API de `users-service`** vía HTTP en lugar de llamar directamente una función local.

## 🧱 Principios básicos del paradigma distribuido

| Principio                           | Descripción                                                          |
| ----------------------------------- | -------------------------------------------------------------------- |
| **Separación de responsabilidades** | Cada servicio cubre una parte funcional concreta del negocio.        |
| **Comunicación mediante red**       | Los servicios interactúan con APIs o colas de mensajes.              |
| **Despliegue independiente**        | Se puede actualizar o reiniciar un servicio sin afectar a los demás. |
| **Escalabilidad selectiva**         | Solo se escalan los componentes que lo requieren.                    |
| **Resiliencia**                     | Un fallo en un servicio no debería detener todo el sistema.          |

---

## 🧠 Beneficios clave

| Beneficio                 | Explicación                                                        |
| ------------------------- | ------------------------------------------------------------------ |
| **Escalado granular**     | Cada componente puede ajustarse según su carga de trabajo.         |
| **Desarrollo paralelo**   | Distintos equipos pueden trabajar en servicios independientes.     |
| **Mantenibilidad**        | Código más acotado, fácil de probar y desplegar.                   |
| **Evolución tecnológica** | Cada servicio puede usar su propio stack (Node.js, Python, Go...). |

---

## ⚠️ Retos de las arquitecturas distribuidas

Aunque solucionan problemas del monolito, también introducen nuevos desafíos:

| Desafío                      | Ejemplo                                                                               |
| ---------------------------- | ------------------------------------------------------------------------------------- |
| **Latencia de red**          | Las llamadas HTTP son más lentas que las llamadas locales.                            |
| **Gestión de errores**       | Si un servicio no responde, hay que manejar timeouts y reintentos.                    |
| **Coherencia de datos**      | Cada servicio puede tener su propia base de datos → surge la “consistencia eventual”. |
| **Monitoreo y trazabilidad** | Se necesitan herramientas centralizadas de observabilidad.                            |

---

## 🔍 Analogía docente

> Imagina una empresa:
>
> * En un monolito, **todo el personal trabaja en una misma oficina**.
> * En una arquitectura distribuida, **cada departamento tiene su propia sede**, pero todos **se coordinan mediante comunicaciones**.

Esta división permite crecer más rápido, pero requiere **organización, normas y canales eficientes**.

---

## 💬 Resumen

| Concepto     | Monolito            | Arquitectura distribuida |
| ------------ | ------------------- | ------------------------ |
| Despliegue   | Uno solo            | Múltiples independientes |
| Comunicación | Interna (funciones) | Externa (HTTP, eventos)  |
| Escalado     | Global              | Granular                 |
| Fallos       | Propagan fácilmente | Aislados por servicio    |
| Tecnología   | Uniforme            | Heterogénea posible      |

---

## 🔚 Conclusión

La arquitectura distribuida es la base conceptual sobre la que se construyen los **microservicios**: una evolución del paradigma que combina **modularidad, independencia y resiliencia**.
En el siguiente módulo comenzaremos a **formalizar el patrón de microservicios**, viendo su estructura, componentes y buenas prácticas.

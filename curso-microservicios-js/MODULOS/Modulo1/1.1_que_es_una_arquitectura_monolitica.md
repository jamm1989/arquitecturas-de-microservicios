# 1.1 – ¿Qué es una arquitectura monolítica?

---

## 🎯 Objetivo

Comprender el concepto de **arquitectura monolítica**, sus características principales y por qué tradicionalmente ha sido el punto de partida para la mayoría de aplicaciones antes de la adopción de los microservicios.

---

## 🧱 Definición general

Una **arquitectura monolítica** es un modelo en el que **toda la lógica de la aplicación se encuentra empaquetada en un único bloque de ejecución**.
Esto significa que:

* El **backend, frontend y lógica de negocio** forman parte del mismo proceso.
* Existe **una sola base de datos compartida**.
* Se genera **un único artefacto desplegable** (por ejemplo, un archivo `.jar`, `.war`, o una app Node ejecutada en un único servidor).

En otras palabras, **todas las funciones de la aplicación viven juntas y se despliegan juntas**.

---

## 🧩 Estructura típica

Una aplicación monolítica suele tener una estructura similar a la siguiente:

```
📦 app/
 ├── controllers/
 ├── services/
 ├── models/
 ├── routes/
 └── app.js
```

* `controllers/`: gestionan las peticiones HTTP
* `services/`: contienen la lógica de negocio
* `models/`: interactúan con la base de datos
* `app.js`: punto de entrada único

**Ejemplo conceptual en Node.js (monolito simple):**

```js
import express from 'express';
const app = express();

const users = [{ id: 1, name: "Ana" }, { id: 2, name: "Luis" }];
const orders = [{ id: 100, userId: 1, total: 59.90 }];

// Endpoints mezclados en la misma app
app.get('/users', (req, res) => res.json(users));
app.get('/orders', (req, res) => res.json(orders));

app.listen(3000, () => console.log('Servidor monolítico en puerto 3000'));
```

➡️ Todo el sistema se ejecuta **en un único servidor y proceso**.
Si una parte falla o necesita escalar, **afecta a todo el conjunto**.

---

## ⚙️ Características principales

| Aspecto                          | Descripción                                                                             |
| -------------------------------- | --------------------------------------------------------------------------------------- |
| **Despliegue único**             | Toda la aplicación se construye y despliega como una sola unidad.                       |
| **Base de datos común**          | Todos los módulos comparten las mismas tablas y esquemas.                               |
| **Comunicación interna directa** | Las funciones se invocan como métodos o clases dentro del mismo proceso.                |
| **Mantenimiento centralizado**   | Actualizar o depurar implica volver a desplegar el bloque completo.                     |
| **Dependencias compartidas**     | Las librerías son comunes a toda la app, lo que puede generar conflictos con el tiempo. |

---

## 💬 Ventajas del enfoque monolítico

* Simplicidad inicial en desarrollo y despliegue.
* No requiere infraestructura distribuida (ideal para prototipos o proyectos pequeños).
* Pruebas locales sencillas (todo en un solo proceso).
* Depuración y logging centralizados.

---

## ⚠️ Desventajas

| Desventaja                               | Impacto                                                                              |
| ---------------------------------------- | ------------------------------------------------------------------------------------ |
| **Dificultad para escalar parcialmente** | Si un módulo necesita más recursos, se debe escalar toda la app.                     |
| **Acoplamiento fuerte**                  | Los módulos dependen unos de otros, dificultando la evolución independiente.         |
| **Tiempo de despliegue largo**           | Cada cambio requiere recompilar y desplegar todo el bloque.                          |
| **Complejidad en equipos grandes**       | Los desarrolladores pisan código de otros, y los ciclos de desarrollo se ralentizan. |
| **Alta dependencia tecnológica**         | Migrar de framework o base de datos es costoso.                                      |

---

## 🧠 Reflexión

> El monolito **no es malo por sí mismo**.
> Es un **modelo adecuado para aplicaciones pequeñas o de bajo tráfico**, o cuando se requiere velocidad inicial de desarrollo.
> El problema surge **cuando crece el tamaño, el equipo y la necesidad de escalar o evolucionar módulos de forma independiente**.

---

## 🔍 Conclusión

Una **arquitectura monolítica** es un único bloque funcional que facilita el inicio de un proyecto, pero **limita la evolución, escalabilidad y resiliencia** a largo plazo.
Este modelo sirve como base para entender por qué surgió la necesidad de **dividir las aplicaciones en microservicios**, tema que se abordará en el siguiente punto (**1.2 – Limitaciones comunes**).
